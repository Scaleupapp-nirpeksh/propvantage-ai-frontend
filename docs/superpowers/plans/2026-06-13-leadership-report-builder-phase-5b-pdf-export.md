# Leadership Report Builder — Phase 5b (PDF Export) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let a viewer download the public report as a clean PDF.

**Architecture:** Native browser **print-to-PDF** — a `@media print` stylesheet isolates the report content, and a "Download PDF" button on the public report page triggers `window.print()`. No new dependencies, vector fidelity (charts/text stay crisp). Frontend-only.

**Why not server-side / auto-attach:** Auto-attaching a high-fidelity PDF to the stakeholder email would require headless Chromium (puppeteer/playwright ≈ 280MB + a chromium download) rendering the public page server-side — a heavy, deploy-fragile addition for EC2. It is **deliberately out of scope here** (documented at the end as an opt-in follow-up). The stakeholder email already links to the page; viewers download the PDF themselves.

**Tech Stack:** React 18, MUI v5, CSS `@media print`.

**Repo for ALL tasks:** `/Users/nirpekshnandan/My Products/propvantage-ai-frontend`. Paths relative to that root.

**Depends on:** Phase 2 `src/pages/public/PublicReportPage.js` (the viewing state).

   Isolates the report content for browser "Save as PDF". */
@media print {
  body * { visibility: hidden; }
  #report-print-root, #report-print-root * { visibility: visible; }

    position: absolute;
    left: 0;
    top: 0;

    margin: 0;

  /* Avoid awkward breaks inside a block/card */
  .report-block { break-inside: avoid; page-break-inside: avoid; }

## Optional follow-up (NOT built here): server-side PDF + email attachment

If auto-attaching a PDF to the stakeholder email is later required:
1. Add `puppeteer` (or `@sparticuz/chromium` + `puppeteer-core` for Lambda/EC2 size limits).
2. A backend service renders the public URL (`/r/:slug`) to a PDF buffer with a temporary internal access token (so the headless browser bypasses the email gate).
3. `deliveryService` attaches the buffer to `sendEmail` (nodemailer `attachments: [{ filename, content }]`).
This adds ~280MB + a chromium runtime and real deploy complexity on EC2 — evaluate against the value before committing to it.
